# 02 · 产品需求文档（PRD）

> 关联文档：[UX 设计](./03-ux-conversational-commerce.md) · [AI 引擎](./05-ai-engine.md) · [供给侧](./06-supply-side.md) · [路线图](./10-roadmap.md)

本文定义 Zeno 的用户、用户故事与功能需求，覆盖**需求端**与**供给端**两侧。功能用 **MoSCoW**（Must / Should / Could / Won't-now）标注优先级，并标注归属阶段（P0=MVP，P1，P2，详见 [路线图](./10-roadmap.md)）。

---

## 1. 用户画像（Personas）

### 需求端（买家）

**P1 · 决策疲惫的务实派 · 林悦（🇨🇳，32，上海，产品经理）**
- 痛点：要买的东西多、没空研究参数、怕踩坑、讨厌被直播话术裹挟。
- 期待："告诉我哪个适合我、为什么，别让我自己刷三小时。"
- 行为：愿意描述需求、愿意比价、看重真实口碑。

**P2 · 参数恐惧的赠礼者 · Sarah（🇺🇸，41，奥斯汀，老师）**
- 痛点：给家人挑礼物/电器，不懂参数，怕买错丢面子。
- 期待："我说用途和预算，你帮我选一个不会出错的。"
- 行为：信任评价与退货政策，决策慢、怕风险。

**P3 · 极致比价的精算师 · 王磊（🇨🇳，27，成都，自由职业）**
- 痛点：同款全网差价大、优惠玩法复杂、算不清到手价。
- 期待："同样的东西，哪里买、配什么券，到手最便宜。"
- 行为：高频、价格极敏感、愿意为返利停留。

**P4 · 高客单的专家型买家 · David（🇺🇸，36，西雅图，工程师）**
- 痛点：买高客单数码/户外，有明确硬约束，受不了泛泛推荐。
- 期待："基于这些硬指标做客观对比，告诉我取舍。"
- 行为：问得细、追问理由、看重中立与数据。

### 供给端（商家）

**S1 · CPS 商品提供方（联盟侧，阶段一）**
- 即已在淘宝/京东/Amazon 等并加入联盟的商家。Zeno 通过联盟接入其商品，无需其单独对接。
- 诉求：被精准曝光、带来真实成交。

**S2 · 0 佣金入驻的中小商家（阶段二）**
- 痛点：在货架平台被高佣金、高买量成本挤压，难被对的人看到。
- 期待："0 佣金、把我的好商品送到真有需求的人面前。"
- 诉求：低成本获客、精准流量、有用的经营工具。

**S3 · 寻求中立曝光的品牌方（阶段二）**
- 期待：在一个"按需求匹配、不靠竞价"的渠道获得高质量曝光与转化。

## 2. 需求端功能需求

### 2.1 核心对话购物（Must · P0）
| ID | 用户故事 | 验收要点 |
|----|----------|----------|
| D-01 | 作为买家，我能用自然语言描述需求（含场景、约束、预算、隐性偏好），让 Zeno 理解 | 能从一句话抽取结构化约束；能识别隐性需求（"久站不累"→缓震/支撑） |
| D-02 | 当我的需求模糊时，Zeno 主动问**最关键的 1–2 个澄清问题**，而非一堆表单 | 澄清问题按"信息增益"排序；不超过必要轮次；可跳过 |
| D-03 | Zeno 跨品牌/跨平台**收窄到 2–4 个**最适合的候选，而不是甩一堆结果 | 默认不超过 4 个；每个有明确推荐理由 |
| D-04 | 每个推荐都**解释"为什么是它"以及取舍**，并可溯源（参数/口碑出处） | 理由对应到用户约束；标注不确定项；不臆造参数 |
| D-05 | 我能对候选发起**结构化对比**（关键维度并排） | 动态生成对比表；维度由需求决定 |
| D-06 | 我能在对话中**追问、改约束、换方向**，Zeno 保持上下文 | 多轮一致；改约束后结果实时更新 |

### 2.2 信任与透明（Must · P0）
| ID | 用户故事 | 验收要点 |
|----|----------|----------|
| D-07 | 我能看到 Zeno 的**利益披露**（含联盟返佣关系） | 清晰标注"含返佣链接"；不影响排序的承诺可见 |
| D-08 | Zeno 给出**真实、可购买**的价格与库存，并指明购买渠道 | 价格/库存经校验；失效不展示或明确标注（[05](./05-ai-engine.md)） |
| D-09 | 我能看到**到手价**（含可叠加的券/返利），中美各按本地玩法 | 中国体现券后/返利；美国体现 coupon/cashback |

### 2.3 个性化与记忆（Should · P1）
| ID | 用户故事 | 验收要点 |
|----|----------|----------|
| D-10 | Zeno 记住我的长期偏好（品牌倾向、预算带、忌讳、尺码/肤质等），跨会话复用 | 可查看/编辑/删除记忆（[07](./07-data-model.md)、[09](./09-trust-safety-compliance.md)） |
| D-11 | Zeno 基于我的历史，主动避免重复推荐我已拒绝/已买的 | 拒绝信号被纳入排序 |
| D-12 | 我能保存"心愿/待决策"，Zeno 在降价或更优替代出现时提醒 | 价格/替代品监控 + 通知 |

### 2.4 履约与售后（Should · P1 / Could · P2）
| ID | 用户故事 | 优先级 |
|----|----------|--------|
| D-13 | 我能**一键跳转**到购买渠道完成下单（带正确归因） | P0（跳转式） |
| D-14 | 我能在 **Zeno 内直接下单**（agentic checkout / ACP），不跳出 | P2 |
| D-15 | 下单后我能在 Zeno 里查物流/发起退换的入口 | P2 |

### 2.5 多模态（Could · P1/P2）
| ID | 用户故事 | 优先级 |
|----|----------|--------|
| D-16 | 我能发图/截图让 Zeno 理解（"找类似这个的""这件配什么"） | P1 |
| D-17 | 我能语音说需求 | P2 |

## 3. 供给端功能需求

### 3.1 阶段一：CPS 接入（Must · P0，对商家无感）
| ID | 需求 | 验收要点 |
|----|------|----------|
| S-01 | 系统能接入多个联盟商品库（淘客/京东/多多进宝/抖音/Amazon/CJ/Rakuten/Impact） | ≥1 中国 + ≥1 美国联盟跑通；详见 [06](./06-supply-side.md) |
| S-02 | 把异构商品**标准化**为统一商品模型并富化 | 字段映射、去重、归一（[07](./07-data-model.md)） |
| S-03 | 生成带正确归因参数的可购买链接，并能对账结算 | PID/SubID/uid 跟单；佣金对账（[06](./06-supply-side.md)、[08](./08-business-model.md)） |
| S-04 | 商品质量校验（价格异常/失效/违禁/低评分过滤） | 上架前过滤规则；可购买性校验 |

### 3.2 阶段二：0 佣金商家入驻（Must · P2）
| ID | 用户故事 | 验收要点 |
|----|----------|----------|
| S-05 | 作为商家，我能自助入驻、上架商品（0 成交佣金） | 入驻流程、资质审核、商品录入/导入 |
| S-06 | 我能用**结构化的需求语言**描述我的商品适合谁，提升被匹配概率 | 引导式商品需求标签；映射到知识图谱（[07](./07-data-model.md)） |
| S-07 | 我能看到**经营看板**（曝光/被匹配/转化/被拒原因），优化供给 | 数据洞察；不暴露他人隐私 |
| S-08 | 我能使用 Zeno 的营销/履约增值服务（按需付费——非佣金变现） | 物流/客服/广告位/数据服务（[08](./08-business-model.md)） |

### 3.3 供给治理（Must，跨阶段）
| ID | 需求 |
|----|------|
| S-09 | 商家/商品分级与信任评分，劣质供给降权或下架 |
| S-10 | 假货/违禁/虚假宣传识别与处置（中美各按本地合规，[09](./09-trust-safety-compliance.md)） |
| S-11 | 价格/库存的近实时同步与失效保护 |

## 5. 非功能需求（NFR）

| 类别 | 要求 |
|------|------|
| **延迟** | 首个有用响应（理解+澄清或初步候选）目标 < 3s；流式输出降低感知延迟（[04](./04-system-architecture.md)、[05](./05-ai-engine.md)） |
| **真实性** | 价格/库存/参数错误率有硬上限；可购买性校验前置；宁缺毋错（[05](./05-ai-engine.md)） |
| **可用性** | 核心对话链路可用性目标 ≥ 99.9% |
| **数据合规** | 中美数据隔离；用户可查看/导出/删除记忆；PIPL/CCPA 就绪（[09](./09-trust-safety-compliance.md)） |
| **可扩展** | 商品库支持千万级 SKU 标准化与近实时更新（[06](./06-supply-side.md)、[07](./07-data-model.md)） |
| **成本** | 单次对话 AI 成本受控；分级模型与缓存（[05](./05-ai-engine.md)、[08](./08-business-model.md)） |
| **可观测** | 全链路指标埋点，支持需求满足率等北极星度量（[11](./11-metrics.md)） |
| **本地化** | i18n、货币、本地优惠玩法、文化表达可配置（[03](./03-ux-conversational-commerce.md)） |
| **无障碍** | 满足基本可访问性（语音/大字/对比度） |

## 6. MVP 的"一句话范围"

> **MVP = 一个品类（如数码）× 一个市场（如中国 CPS）× 一个杀手体验（描述需求 → AI 收窄 → 解释推荐 → 一键履约）+ 真实价格/库存 + 利益披露。**

不在 MVP：0 佣金入驻、站内支付、语音、跨市场。详见 [路线图](./10-roadmap.md)。

## ⚖️ 正反方辩论（Red Team / Blue Team）

> 对本文关键主张做一次对抗性审视——正方力挺、反方进攻，最后给出裁决与"会改变主意的信号"。

**议题：先做需求侧杀手体验、把 0 佣金入驻与站内支付后置，优先级对吗？**

**🟦 正方（支持）**
- 用一个杀手体验先验证核心假设，避免过早做重（入驻 / 支付 / 履约）。
- CPS 跳转履约已足以验证"AI 能否找到对的货"这一最大风险。

**🟥 反方（质疑）**
- 跳转式履约漏斗流失大、归因弱、体验割裂，可能严重低估转化损失。
- 把个性化记忆放到 P1 可能太晚——个性化往往才是留存的真正杠杆。
- MoSCoW 易被需求蔓延侵蚀，"Should"会悄悄变成"Must"。

**⚖️ 裁决**
- MVP 接受跳转流失以换取速度，但必须埋点量化流失；过高则提前引入 ACP 站内直购。
- 若 P0 数据显示记忆是留存杠杆，则把它提前到 P0/P1 交界。
- **Kill 信号：** 跳转流失率高到单位经济为负，且 ACP 也无法缓解。
