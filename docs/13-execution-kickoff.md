> 本文把 [路线图](./10-roadmap.md) 的 **P0 / MVP** 转成可执行的工程计划：第一个垂直切片、技术栈、Sprint backlog、完成标准、立即下一步。**部分默认选择（市场/品类/联盟/栈）需你拍板**，见 §2 与文末。

## 1. 原则：从"设计"到"一条能跑通的细线"

不追求一次做全，而是先打通**一条端到端最薄的垂直切片（thin vertical slice）**：

> **一句真实需求 → 理解 → 检索 → 排序 → 2–4 候选 + 理由 → 带正确归因的购买链接。**

只要这条线在**一个很窄的子品类**上真实跑通（真实商品、真实价格、真实归因），就验证了 Zeno 最大的风险假设："AI 能否从全网找到对的货并被采纳"。其余一切都是在这条线上加厚。

## 3. 第一个垂直切片（End-to-End Thin Slice）

```
用户输入一句需求（如"通勤用、戴久不夹耳、1000内的耳机"）
   │
   ▼  [understand]  LLM 抽取 Need Spec（硬/软/隐性/场景）+ 最小澄清
   ▼  [retrieve]    pgvector 混合检索（结构化过滤 + 向量语义）召回候选
   ▼  [rank]        多目标排序（契合度/质量/到手价/真实可买）→ top 2–4
   ▼  [verify]      可购买性校验（价格/库存真值）
   ▼  [present]     流式返回：候选卡 + 推荐理由 + 利益披露
   ▼  [fulfill]     生成带 SubUnionId 的购买链接 + 记录归因事件
   ▼  [measure]     埋点：采纳/澄清轮次/（后续）后悔率
```

**切片的"最小"边界**：单品类子集、单联盟、跳转履约、无长期记忆（记忆放 Sprint 2+）、无 0 佣金入驻。

## 4. 技术栈建议（可执行，非约束）

为"小团队、AI 重、要流式聊天 + 数据管道"优化：

| 层 | 建议 | 说明 |
|----|------|------|
| **对话应用（前后端）** | **Next.js (TypeScript)** + SSE 流式 | 一种语言起步、流式聊天体验好、部署简单；承载混合 UI（[03](./03-ux-conversational-commerce.md)） |
| **AI 接入** | **Anthropic SDK + Claude 分级**（经薄 AI Gateway 封装） | Haiku 抽取/富化 · Sonnet 主对话 · Opus 难推理（[04 §3](./04-system-architecture.md)、`claude-api` 参考） |
| **统一存储 + 检索** | **Postgres + pgvector**（向量 + 结构化 + 全文一体） | MVP 阶段一个库够用，省运维；后续再上专用向量/图引擎（[07](./07-data-model.md)） |
| **供给管道** | **Python**（联盟接入 + 标准化 + LLM 富化） | 数据/AI 生态最顺手；离线/近线运行，与在线应用解耦（[06 §2.2](./06-supply-side.md)） |
| **可观测/评测** | 结构化日志 + 最小 Eval 脚本（黄金需求集） | 从第一天就量化需求满足率/幻觉率（[11](./11-metrics.md)） |
| **部署** | 单区域起步（按目标市场），IaC 可后补 | 双区域待第二市场（[04 §5](./04-system-architecture.md)） |

> **替代路线**：若团队更偏 AI/数据，可全 Python（FastAPI + 轻前端）。关键是**别在 MVP 过早上微服务/双区域/专用图库**——先把细线跑通。

## 5. 模块 → 代码的最小映射

| 设计模块 | MVP 最小实现 | 暂不做 |
|----------|--------------|--------|
| 需求理解（[05 §1](./05-ai-engine.md)） | 一个 LLM 调用，输出 Need Spec JSON（schema 校验） | 复杂多轮记忆 |
| 检索（[05 §2](./05-ai-engine.md)） | pgvector 结构化过滤 + 向量召回 | 知识图谱推理、全文引擎 |
| 排序（[05 §3](./05-ai-engine.md)） | 规则 + 轻量打分（契合度/到手价/质量），输出理由素材 | 学习型排序模型 |
| 编排（[05 §4](./05-ai-engine.md)） | 直白的 agent loop + 工具调用（understand/retrieve/rank/verify/fulfill） | 复杂分支策略 |
| 富化（[05 §5](./05-ai-engine.md)） | Python 脚本批量富化单子品类（属性/场景/向量） | 全品类、自动化调度 |
| 归因（[06 §2.3](./06-supply-side.md)） | 生成 SubUnionId 链接 + 落库点击事件 | 自动对账（先手动核对） |
| 真实性（[05 §6](./05-ai-engine.md)） | 展示前价格/库存校验 + 不确定标注 | — |

## 6. Sprint Backlog（建议 2 个冲刺，各约 1–2 周）

### Sprint 0 · 地基（让一条假数据的线能跑）
- [ ] 仓库工程化：monorepo（app / pipeline）、lint、CI、env 管理
- [ ] AI Gateway 薄封装（Claude 分级调用 + 流式 + 重试/降级）
- [ ] Postgres + pgvector 起库；`Product` / `Offer` 最小 schema（[07](./07-data-model.md)）
- [ ] 京东联盟接入 spike：能拉到单子品类商品 + 生成带 SubUnionId 的链接
- [ ] 富化脚本 v0：把该子品类商品抽成结构化属性 + 向量（Python）
- [ ] 聊天 UI 骨架：输入框 + 流式输出 + 候选卡组件
- [ ] 埋点骨架：会话/候选/点击事件落库

### Sprint 1 · 打通细线（真实数据 end-to-end）
- [ ] `understand`：一句话 → Need Spec（含隐性推断 + 最小澄清）
- [ ] `retrieve` + `rank`：召回 → 2–4 候选 + 理由素材
- [ ] `verify`：展示前价格/库存校验；失效则替换/标注
- [ ] `present`：候选卡 + 推荐理由 + **利益披露**（含返佣标注，[09](./09-trust-safety-compliance.md)）
- [ ] `fulfill`：跳转购买 + 归因事件闭环
- [ ] Eval v0：20–50 条黄金需求集 + 需求满足率/幻觉率离线打分（[11](./11-metrics.md)）
- [ ] 内部可用：团队能对真实需求得到真实可买的推荐

## 7. 定义完成（Definition of Done）

MVP thin slice 视为完成，当：
1. 对该子品类的**真实自然语言需求**，能返回 **2–4 个真实、可买、价格正确**的候选；
2. 每个候选有**对应约束的推荐理由** + **利益披露**；
3. 购买链接带**正确归因**，点击事件可追踪；
4. **幻觉率**（价格/参数）与**需求满足率**有埋点与基线读数；
5. 全链路**首个有用响应 < 3s**（流式）。

> 通过后即对照 [路线图 P0 门槛指标](./10-roadmap.md) 决定是否进入 P1。

## ⚖️ 正反方辩论（Red Team / Blue Team）

> 对本文关键主张做一次对抗性审视——正方力挺、反方进攻，最后给出裁决与"会改变主意的信号"。

**议题：现在就动手建 MVP thin slice（窄品类 + 单联盟 + 跳转履约），是对的推进方式吗？**

**🟦 正方（支持）**
- 端到端细线能最快验证最大风险假设，胜过继续纸面完善。
- 窄品类 + 单联盟把不确定性压到最小，富化/检索/排序都能做透。

**🟥 反方（质疑）**
- 选错市场/品类/联盟，细线跑通也可能是"伪验证"。
- 过早写代码可能锁定错误抽象；联盟资质审批存在不可控周期。
- 单子品类样本太小，需求满足率结论的外推性存疑。

**⚖️ 裁决**
- 先确认 §2 配置再动手；工程结构保持"可替换市场/联盟"的抽象，不写死。
- 联盟 spike 与工程地基并行，吸收审批周期；细线跑通后立即用第二个子品类验证外推。
- **Kill 信号：** 细线在两个子品类上的需求满足率都不及货架搜索基线，且富化无法规模化。
