# 04 · 系统架构

> 关联文档：[AI 引擎](./05-ai-engine.md) · [供给侧](./06-supply-side.md) · [数据模型](./07-data-model.md) · [合规](./09-trust-safety-compliance.md)

本文描述 Zeno 的系统总览、核心服务拆分、AI Gateway、数据管道与中美双区域部署。架构目标：**低延迟的对话体验、真实可买的供给、可演进到双市场与自有供给、合规可隔离。**

## 2. 核心服务拆分

| 服务 | 职责 | 关键技术取向 |
|------|------|--------------|
| **BFF / API Gateway** | 鉴权、限流、区域路由、流式传输、多端适配 | SSE/WebSocket；CN/US 入口隔离 |
| **对话编排 Orchestrator** | Agent 主循环：维护会话状态、决定下一步动作（澄清/检索/对比/推荐）、调度工具、组装混合 UI | 有状态会话；工具编排；详见 [05](./05-ai-engine.md) |
| **需求理解 Understanding** | 抽取结构化约束、识别隐性偏好、意图分类、澄清问题生成 | LLM 抽取 + 约束 schema（[07](./07-data-model.md)） |
| **检索 Retrieval** | 混合检索：向量语义 + 结构化过滤 + 全文 + 知识图谱推理 | 向量库 + 倒排 + KG；召回-精排两段 |
| **排序 Ranking** | 多目标排序（相关性/质量/价格/信任/转化），输出 2–4 候选 + 理由素材 | 多目标、可解释、不接受竞价信号 |
| **记忆/画像 Memory** | 长期偏好、拒绝信号、尺码/肤质、心愿与监控 | 用户可查/改/删（[09](./09-trust-safety-compliance.md)） |
| **履约/归因 Fulfillment & Attribution** | 生成带归因的购买链路、对账、（未来）站内 ACP 下单、订单/物流/售后入口 | PID/SubID/uid；ACP/SPT（[06](./06-supply-side.md)、[08](./08-business-model.md)） |
| **供给数据管道 Supply Pipeline** | 联盟接入、标准化、LLM 富化、去重、质检、索引、近实时同步 | 批 + 流；详见 [06](./06-supply-side.md) |
| **商家平台 Merchant（P2）** | 0 佣金入驻、商品录入、经营看板、增值服务 | 自助化；数据洞察 |
| **AI Gateway** | 统一多模型调用、路由、缓存、降级、成本与配额、安全过滤 | 见 §3 |
| **可观测 + 评测** | 全链路 tracing、指标、在线/离线 Eval（需求满足率等） | 见 [11](./11-metrics.md) |

> 拆分原则：**沿"对话主链路"做强一致与低延迟，沿"供给管道"做高吞吐与最终一致。** 两者通过索引解耦。

## 3. AI Gateway（模型层）

对话体验的智能全部经由 **AI Gateway** 统一访问，模型可插拔、可分级、可降级。

**设计要点**
- **模型无关 + 默认最强**：网关屏蔽供应商差异；默认采用能力最强的 Claude 模型族，按任务**分级**以平衡质量/延迟/成本：
  - **重推理**（复杂需求理解、取舍解释、争议对比）→ 旗舰级（如 Claude Opus 4.8，`claude-opus-4-8`）。
  - **主对话/编排**（多数轮次）→ 均衡级（如 Claude Sonnet 4.6，`claude-sonnet-4-6`）。
  - **高频轻任务**（约束抽取、富化、改写、分类）→ 高速级（如 Claude Haiku 4.5，`claude-haiku-4-5-20251001`）。
  > 模型 ID 以接入时官方为准；网关支持灰度与 A/B 切换。集成细节遵循 Anthropic SDK 规范（参见 `claude-api` 参考）。
- **工具使用（Tool Use）**：编排器以工具调用方式驱动检索/对比/价格校验等，模型只决定"做什么"，真实数据由工具提供——这是**抗幻觉**的关键（[05](./05-ai-engine.md)）。
- **流式优先**：首 token 尽快返回，边推理边渲染，降低感知延迟。
- **缓存与复用**：系统提示、商品上下文、相似查询做提示缓存与结果缓存，降本增速。
- **降级与熔断**：高负载/故障时自动降级模型或退化为结构化检索，保证可用。
- **成本与配额**：按会话计量，预算护栏（[08](./08-business-model.md)）。
- **安全过滤**：输入输出经安全策略（prompt 注入防护、有害内容、合规用语，[09](./09-trust-safety-compliance.md)）。

## 4. 数据管道（供给侧）

两条路径汇入统一商品索引（详见 [06](./06-supply-side.md)、[07](./07-data-model.md)）：

1. **批量接入**：联盟商品库全量/增量导入（定时拉取或 feed）。
2. **近实时同步**：价格/库存/失效的高频更新（流式），保证"真实可买"。

**处理阶段**：`接入 → 清洗 → 标准化(字段映射) → LLM 富化(抽属性/场景/人群) → 去重(跨平台同款合并) → 质检(违禁/异常价/低质过滤) → 索引(向量+结构+全文+KG)`。

**一致性策略**：商品**内容**最终一致即可；商品**价格/库存/可购买性**要求新鲜度高——展示前经"可购买性校验"二次确认（[05](./05-ai-engine.md) §真实性）。

## 5. 中美双区域部署

> 原则：**一个核心、两套部署、数据隔离、合规分支。**

```
        ┌───────────────────────┐        ┌───────────────────────┐
        │      区域 CN 🇨🇳        │        │      区域 US 🇺🇸        │
        │  境内云 · 数据不出境   │        │  美国云                │
        │  PIPL/电商法/算法规定  │        │  CCPA/州法/FTC 披露    │
        │  供给:淘客/京东/多多/抖│        │  供给:Amazon/CJ/Rakuten│
        │  音; 微信/支付宝;小程序│        │  /Impact; 卡组织/ACP   │
        └───────────┬───────────┘        └───────────┬───────────┘
                    │   仅共享"非个人数据"的核心资产    │
                    └──────────────┬───────────────────┘
                                   ▼
                ┌──────────────────────────────────┐
                │  跨区共享(脱敏/聚合，不含个人数据) │
                │  · 模型与提示/编排逻辑             │
                │  · 商品知识图谱本体(schema/类目)   │
                │  · 评测集与质量基线               │
                └──────────────────────────────────┘
```

- **用户个人数据严格按区域驻留**，不跨境（满足 PIPL 数据出境 / 美国州法），详见 [09](./09-trust-safety-compliance.md)。
- **可复用的是"非个人"的核心资产**：编排逻辑、知识图谱本体、评测基线、模型选择策略。
- **供给、支付、履约、合规用语**为区域本地化分支。
- 部署上 CN 使用境内云与境内可用的模型/网关方案，US 使用对应区域服务，二者通过**接口契约**保持同构，而非共享存储。

## 6. 关键质量属性如何被架构支撑

| 质量属性 | 架构手段 |
|----------|----------|
| **低延迟** | 流式、提示/结果缓存、召回-精排两段、分级模型、就近部署 |
| **真实可买** | 价格/库存近实时同步 + 展示前可购买性校验 + 工具取真值（非模型臆造） |
| **可解释** | 排序输出"理由素材"，编排器据此生成可溯源解释 |
| **中立** | 排序不接受竞价信号；归因与排序解耦（[08](./08-business-model.md)、[09](./09-trust-safety-compliance.md)） |
| **可演进** | 供给管道与对话链路经索引解耦；商家直连作为新数据源接入同一管道 |
| **合规隔离** | 双区域数据驻留 + 本地化合规分支 |
| **可观测/可评测** | 全链路 tracing + 在线/离线 Eval 闭环（[11](./11-metrics.md)） |

## 7. 技术栈建议（非约束，按团队取舍）

- **在线服务**：Go / Rust（网关、检索、归因等低延迟服务）+ Python（AI/数据/富化）。
- **对话编排**：事件驱动、有状态会话存储（Redis/状态库）。
- **检索**：向量库（如 pgvector / 专用向量引擎）+ 倒排全文 + 图存储（KG）。
- **数据管道**：流（Kafka/类似）+ 批（编排调度）；湖仓存原始与富化数据。
- **AI**：Anthropic SDK 接入 Claude 模型族（经 AI Gateway），遵循 `claude-api` 参考。
- **部署**：CN/US 双区域，IaC 管理，可观测（Tracing/Metrics/Logging）一体化。

> 栈的选择服从两条主线：**对话链路求快与稳，供给管道求吞吐与质量。**
