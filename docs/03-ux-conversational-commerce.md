# 03 · 对话式购物体验设计

> 关联文档：[PRD](./02-product-requirements.md) · [AI 引擎](./05-ai-engine.md) · [信任与合规](./09-trust-safety-compliance.md)

Zeno 的体验目标：**让用户花最少的认知与时间，做出自己信任的购买决定。** 对话不是为了"聊天"，而是为了**高效收窄**。

---

## 1. 设计原则

1. **对话用于收窄，不用于闲聊（Conversation to converge）**——每一轮都应让用户离"对的决定"更近。
2. **少即是多（Fewer, better）**——默认给 2–4 个候选，不给一墙结果。选择越少，决策越轻。
3. **永远讲理由（Always explain）**——每个推荐都回答"为什么是它 / 它的取舍"。理由可溯源。
4. **混合 UI（Hybrid UI）**——文字对话 + 动态生成的卡片 / 对比表 / 筛选 chip，各取所长。该用结构化就用结构化，别硬聊。
5. **用户随时可掌舵（User in control）**——可改约束、可追问、可跳过澄清、可"直接给我看选项"。
6. **真实可买（Real & buyable）**——展示的价格、库存、链接都真实可成交，失效即标注。
7. **信任前置（Trust by design）**——利益披露、不确定性、口碑出处都显性化，不藏。

---

## 2. 核心流程：从模糊需求到下单

```
① 表达需求        用户用自然语言说需求（可模糊、可带场景/约束/预算）
      │
② 理解 + 澄清     抽取结构化约束；仅在必要时问 1–2 个高增益澄清问题
      │            （能猜的就猜，并把假设显性说出来让用户纠正）
      │
③ 收窄 + 推荐     跨平台检索→多目标排序→给 2–4 个候选，每个带推荐理由
      │
④ 对比 + 追问     用户并排对比、追问取舍、改约束；结果实时更新
      │
⑤ 决定           用户选定；Zeno 给出到手价、利益披露、风险提示
      │
⑥ 履约           跳转购买（带归因）→ 未来站内 ACP 下单
      │
⑦ 记忆 + 跟进     记录偏好与拒绝信号；可设降价/替代品提醒
```

### 关键设计选择
- **能猜则猜，假设外显**：与其问"你预算多少？要不要降噪？"，不如先给出"我先按 800 内、要降噪来挑，你可以随时改"。把澄清成本降到最低。
- **澄清问题按信息增益排序**：只问那些"答案会显著改变推荐"的问题，最多 1–2 个。详见 [AI 引擎 §澄清策略](./05-ai-engine.md)。
- **始终给"逃生口"**：用户随时可说"别问了，直接给我看"。

## 3. 对话设计模式（Conversation Patterns）

| 模式 | 何时用 | 示例 |
|------|--------|------|
| **澄清（Clarify）** | 关键约束缺失且影响大 | "主要是通勤用还是健身用？这会影响我推降噪还是防水。" |
| **假设外显（Assume & confirm）** | 能合理推断 | "我先假设预算 800 内、要好降噪，按这个挑了——不对随时说。" |
| **收窄（Narrow）** | 候选过多 | "全网符合的有很多，我帮你压到这 3 个最值得看的。" |
| **推荐 + 理由（Recommend & justify）** | 给候选 | "选它，因为①续航 60h 满足你≥2 天；②半开放不夹耳。代价是降噪比入耳弱一档。" |
| **对比（Compare）** | 用户在候选间犹豫 | 动态对比表：续航/降噪/佩戴/到手价 并排 |
| **取舍坦白（Trade-off honesty）** | 没有完美解 | "这个价位无法同时要顶级降噪和超轻佩戴，你更在意哪个？" |
| **反向推荐（Talk you out of it）** | 用户选了不适合的 | "说实话这个对你的'久戴不疼'诉求不太合适，要不要看这个？"（中立信任的体现） |
| **兜底（Graceful fallback）** | 无合适商品 | "目前没有同时满足全部硬约束的，最接近的是这两个，差在 X。" |

## 4. 混合 UI：对话 + 动态组件

纯文字聊不适合展示商品。Zeno 在对话流中**动态渲染**结构化组件：

- **商品候选卡**：图、名、到手价、一句话推荐理由、关键匹配点 chip、利益披露标记、购买按钮。
- **对比表**：维度由当前需求动态决定（不是固定模板），高亮各自优劣。
- **约束 chip 区**：把已识别的约束做成可点的 chip（"预算≤800""非入耳""降噪"），用户一点即改，比打字快。
- **澄清快选**：澄清问题给 2–4 个快捷选项 + "其它/跳过"。
- **到手价明细**：券 / 返利 / 满减如何叠加（中美本地化展示）。
- **决策摘要**：选定后给"为你总结：你要 X，我推荐 Y，因为 Z，到手 ¥N，注意 W"。

> 设计取向：**默认对话驱动，按需结构化**。用户主导节奏，AI 在合适时机"长出"UI。

---

## 5. 多模态

- **图片输入（P1）**：上传/截图 → 识别品类与属性 → "找类似""找更便宜的同款""这件配什么"。
- **语音输入（P2）**：移动端口语化表达需求。
- **图片输出**：候选图、对比图、（美国）虚拟试穿类能力可后续接入。

---

## 6. 信任设计（Trust UX）

信任是 Zeno 的立身之本，必须在体验层显性化：

1. **利益披露**：含返佣的链接明确标注；并承诺"返佣高低不影响推荐排序"，该承诺可见可查。
2. **不确定性显性**：参数没核实就标"待核实"，不臆造；口碑结论标出处与样本。
3. **可解释**：每个推荐的理由对应到用户的具体约束，点开可看依据。
4. **中立证据**：敢于推荐"去别处买更便宜"、敢于"劝退不合适的选择"——这是信任的最强信号。
5. **隐私可感知**：记忆可查看 / 编辑 / 删除，用户清楚 Zeno 记住了什么、为什么（[09](./09-trust-safety-compliance.md)）。

## 7. 中美本地化差异

| 维度 | 🇨🇳 中国 | 🇺🇸 美国 |
|------|---------|---------|
| **语气** | 高效、接地气、懂行的朋友；可适度玩梗 | 简洁、客观、可信赖的顾问 |
| **价格表达** | 券后价 / 返利 / 满减叠加、到手价是刚需 | coupon / cashback / 总价透明、税与运费 |
| **信任锚点** | 销量、真实买家秀、比价 | 评分数量、退货政策、品牌口碑 |
| **履约预期** | 极快物流、多渠道比价 | Prime 时效、退货便利 |
| **入口习惯** | 微信生态 / App / 小程序 | Web / App / 浏览器助手 |
| **合规表达** | 广告法用词红线（避免"最/第一"等绝对化） | FTC 联盟披露、AI 披露需显著 |

> 核心对话引擎统一，**语气、价格玩法、信任锚点、合规用语在本地化层处理**。详见 [系统架构](./04-system-architecture.md) 与 [合规](./09-trust-safety-compliance.md)。

## 8. 一个端到端示例（中国 / 数码）

```
用户：耳机推荐一下，预算一千以内，戴久了别夹耳朵

Zeno：好。我先按「≤¥1000、长时间佩戴舒适」来挑。再确认一个关键点——
      [通勤要降噪] [主要在家/安静] [运动防水]   （← 快选 chip）

用户：[通勤要降噪]

Zeno：这三个最合你（半开放/头戴，避开你怕的"夹耳"）：
      ┌─────────────────────────────────────────────┐
      │ A  头戴·主动降噪   续航40h   到手 ¥899        │
      │    ✓久戴不夹耳 ✓通勤降噪够用  代价:略重       │
      │ B  开放式·骨传导   续航10h   到手 ¥749        │
      │    ✓最不闷耳 ✓运动友好   代价:降噪弱          │
      │ C  头戴·旗舰降噪   续航30h   到手 ¥999        │
      │    ✓降噪最强   代价:夹耳感比A略明显           │
      └─────────────────────────────────────────────┘
      含返佣链接 · 返佣不影响以上排序   [对比这三个]

用户：A 和 C 有啥区别

Zeno：[对比表] 降噪：C 更强一档；佩戴：A 更不夹耳（更贴合你的诉求）；
      续航：A 40h > C 30h；到手价：A ¥899 < C ¥999。
      → 你最在意"别夹耳"，我更建议 A；除非你常在很吵的地铁、愿为降噪牺牲点舒适选 C。

用户：那就 A

Zeno：为你总结：你要「≤¥1000、久戴不夹耳、通勤降噪」→ 选 A，
      到手 ¥899（已含 ¥100 券）。下单后我帮你记住你偏好"半开放、不夹耳"。
      [去下单（含返佣）]
```

> 这个例子体现了全部原则：假设外显、最小澄清、收窄到 3 个、可解释、对比、坦白取舍、反映用户真实诉求、利益披露、记忆。

## ⚖️ 正反方辩论（Red Team / Blue Team）

> 对本文关键主张做一次对抗性审视——正方力挺、反方进攻，最后给出裁决与"会改变主意的信号"。

**议题："对话 + 混合 UI、2–4 候选、最小澄清"是最优体验范式吗？**

**🟦 正方（支持）**
- 减少认知负荷、可解释、可信任；少选项降低决策瘫痪。
- 混合 UI 在该结构化时结构化，比纯聊天高效。

**🟥 反方（质疑）**
- 重度比价用户（尤其中国）可能想"看更多、自己比"，2–4 个显得信息不足或被操纵。
- 对话比点选慢，高频场景反而低效；"假设外显"若假设常错会显得 AI 不懂我。
- 混合 UI 工程复杂、迭代慢。

**⚖️ 裁决**
- 默认少而精，但永远提供"展开更多 / 进入比价"逃生口，照顾比价人群。
- 对话非唯一交互，允许快捷点选；用"纠偏轮次"监控假设质量。
- **Kill 信号：** 用户大量绕过对话直奔"给我看全部"，说明对话范式不被接受。
