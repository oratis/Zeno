# 05 · AI 核心引擎

> 关联文档：[系统架构](./04-system-architecture.md) · [数据模型](./07-data-model.md) · [UX](./03-ux-conversational-commerce.md) · [度量](./11-metrics.md)

AI 引擎是 Zeno 的大脑，负责把"一句模糊的需求"变成"少数几个最适合的、可解释的、真实可买的候选"。它由五个子系统组成：**需求理解 → 检索 → 排序 → 对话编排 → 商品理解/富化**，并由**评测与真实性保障**贯穿。

---

## 1. 需求理解（Demand Understanding）

把自然语言需求解析为可计算的 **需求规格（Need Spec）**。

**抽取内容**
- **意图/品类**：要买什么（含跨品类，如"送礼"可跨多个品类）。
- **硬约束（hard）**：预算上限、尺寸、兼容性、必须/排除项（"非入耳"）。
- **软偏好（soft）**：风格、品牌倾向、对某维度的相对重视度。
- **隐性需求（implicit）**：从表述推断（"久站不累"→缓震/支撑/重量；"小户型"→尺寸/收纳）。
- **场景/人群（context）**：使用场景、使用者（给妈妈/给娃/自用）、时间（通勤/健身）。

**需求规格（示意）**
```json
{
  "category": "headphones",
  "hard": { "price_max": 1000, "form_factor": { "not": "in_ear" } },
  "soft": { "noise_canceling": 0.8, "comfort_long_wear": 0.9, "portability": 0.5 },
  "implicit": ["avoid_ear_fatigue"],
  "context": { "use_case": ["commute"], "buyer": "self" },
  "missing_high_gain": ["noise_env"]   // 待澄清且高信息增益的点
}
```

**澄清策略（最小必要）**
- 只问 **信息增益高** 的缺失项（答案会显著改变推荐的）。用"预期推荐分布的变化"来估增益。
- 每轮最多 1–2 个；优先**假设外显**（先按合理默认推，再让用户纠正），而非追问。
- 用户可随时"跳过澄清，直接看候选"。详见 [UX §澄清](./03-ux-conversational-commerce.md)。

---

## 2. 检索（Hybrid Retrieval）

从千万级商品中**召回**满足需求的候选集。单一手段都不够，采用**混合检索**：

| 通道 | 作用 | 例子 |
|------|------|------|
| **结构化过滤** | 硬约束精确过滤 | `price ≤ 1000 AND form_factor ≠ in_ear AND in_stock` |
| **向量语义** | 软偏好/场景的语义匹配 | "久戴不累通勤" ↔ 商品富化语义 |
| **全文/关键词** | 精确词、型号、品牌 | 具体型号、兼容机型 |
| **知识图谱推理** | 关系/场景/兼容/替代 | "适合小户型"→收纳类目；"X 的平替"→替代关系 |

**两段式**：先**召回**（高 recall，混合多通道取并集，过硬约束）→ 再交给排序做**精排**。召回阶段宁可多召，不可漏掉真正合适的（漏召直接拉低需求满足率）。

> 富化质量决定检索上限——脏乱原始数据无法语义匹配。见 §5 与 [07](./07-data-model.md)。

---

## 3. 排序（Multi-Objective Ranking）

从召回集里选出**最适合用户的 2–4 个**，并产出"理由素材"。这是 Zeno 区别于货架"按销量/竞价排"的核心。

**多目标信号（按需求动态加权）**
1. **需求契合度（fit）**——对硬约束的满足 + 对软偏好的加权匹配（主导项）。
2. **商品质量（quality）**——真实口碑、退货率、商家信任分（[06](./06-supply-side.md)）。
3. **价格价值（value）**——到手价 / 性价比，而非绝对低价。
4. **真实可买（availability）**——库存、可购买性、履约时效。
5. **个性化（personal）**——用户记忆：偏好、拒绝信号、已购去重（[07](./07-data-model.md)）。

> ❌ **明确排除的信号：商业竞价 / 返佣高低。** 返佣不进入排序，这是中立性的制度性保证（[09](./09-trust-safety-compliance.md)）。

**输出**
- 2–4 个候选；
- 每个候选的**理由素材**（命中了哪些约束、强在哪、代价是什么、不确定项）；
- 多样性约束：候选之间应有**有意义的差异**（不同取舍点），而非近似重复，便于用户对比决策。

**可解释**：排序把"为什么"结构化输出，编排器据此生成自然语言解释并溯源——不是事后编理由。

## 4. 对话编排（Agent Orchestration）

编排器是 Agent 主循环，决定"下一步做什么"，并以**工具调用**驱动真实能力。

**主循环（简化）**
```
loop:
  读取会话状态 + 最新用户输入
  → 更新 Need Spec（调用 理解）
  → 决策下一步:
       若关键约束缺失且高增益 → 生成澄清(或假设外显)
       否则 → 调用 检索 → 调用 排序
  → 调用 价格/库存校验(可购买性)
  → 组装 混合UI 响应(候选卡/对比表/理由/披露)
  → 流式返回; 等待用户(追问/改约束/选定)
  → 若选定 → 履约(归因链路 / ACP)；写入 记忆
```

**工具（Tool Use）清单（示意）**
- `understand(query, history) → NeedSpec`
- `retrieve(NeedSpec) → candidates[]`
- `rank(NeedSpec, candidates, userMemory) → ranked[] + rationale`
- `verify_buyable(productIds) → {price, stock, url}`（真值，防幻觉）
- `compare(productIds, dimensions) → table`
- `memory.read/write(userId, ...)`
- `fulfill(productId) → attributed_link | acp_checkout`

> **原则：模型决定"做什么"，工具提供"真实数据"。** 价格、库存、参数等事实一律来自工具，模型不得臆造——这是抗幻觉的架构性保证。

## 5. 商品理解与富化（Product Enrichment）

把联盟来的**异构、脏乱**原始商品，变成**可推理的结构化知识**。这是把"全网商品"变成"AI 能精准匹配的供给"的关键资产。

**富化产物**
- **标准化属性**：从标题/详情/参数/图片抽取结构化属性（容量、续航、材质、尺寸……）。
- **适用场景/人群**：这个商品适合谁、什么场景（"通勤""小户型""敏感肌"）。
- **口碑提炼**：从评论聚合真实优缺点（带样本量与出处，不臆造）。
- **关系**：同款（跨平台合并）、替代/平替、配件兼容、升级/降级关系。
- **语义向量**：用于检索的多视角嵌入。

**做法**：以高速级 LLM 批量富化（成本敏感），辅以规则与品类模板；产物写入**商品知识图谱**（[07](./07-data-model.md)）。富化是离线/近线大吞吐任务，与在线对话解耦（[04](./04-system-architecture.md)）。

---

## 6. 真实性保障（Factuality / Anti-Hallucination）

**信任的底线：宁可少说，不可说错。** 多层防护：

1. **事实来自工具，不来自模型记忆**：价格/库存/参数/链接均由工具取真值。
2. **可购买性校验前置**：展示候选前二次校验价格与库存，失效则替换或明确标注。
3. **不确定显性化**：未核实的参数标"待核实"；口碑结论标样本量与出处。
4. **引用可溯源**：推荐理由能回链到数据来源。
5. **约束硬校验**：硬约束（预算、排除项）在排序后做规则复核，杜绝"推荐了被排除项"。
6. **输出守卫**：合规用语过滤（中国广告法绝对化用词、夸大功效；美国 FTC 披露），见 [09](./09-trust-safety-compliance.md)。

## 7. 评测体系（如何衡量"最适合"）

"找到最适合的"不是玄学，必须可度量、可回归。

**离线评测**
- **黄金需求集**：人工构造的"需求→理想候选"数据集，覆盖各品类与难度（含模糊、含冲突约束、含无解兜底）。
- **指标**：需求契合度（人工/LLM 评审）、硬约束零违反率、候选多样性、解释忠实度（理由是否真对应数据）、幻觉率（价格/参数错误）。
- **LLM-as-judge + 人工**：用裁判模型规模化打分，人工抽检校准（防裁判偏差）。

**在线评测**
- **需求满足率（Need-Fit Rate）**、采纳率（用户是否选了 Zeno 的推荐）、决策时长、改约束/追问轮次、购买转化、**事后满意/后悔率**（北极星，[11](./11-metrics.md)）。
- A/B 实验平台：模型/排序权重/澄清策略均可实验。

**闭环**：线上 badcase → 回流黄金集 → 回归 → 富化/排序/提示迭代。详见 [11-度量](./11-metrics.md)。

## 8. 端到端数据流（一次对话）

```
用户输入
  └→[理解] NeedSpec(+澄清?)──┐
                              ▼
                         [检索] 混合召回 candidates[]
                              ▼
        userMemory ──→ [排序] 多目标精排 → top 2–4 + 理由素材
                              ▼
                    [verify_buyable] 价格/库存真值校验
                              ▼
                 [编排] 组装混合UI(候选卡/对比/披露) ──流式──→ 用户
                              ▼ (用户选定)
                    [履约] 归因链路/ACP   [记忆] 写偏好/拒绝信号
                              ▼
                         [评测] 埋点 → 需求满足率/后悔率
```

> AI 引擎的所有智能都服务一个目标：**用最少的用户认知，得到用户自己信得过的购买决定。**

## ⚖️ 正反方辩论（Red Team / Blue Team）

> 对本文关键主张做一次对抗性审视——正方力挺、反方进攻，最后给出裁决与"会改变主意的信号"。

**议题：混合检索 + 多目标排序 + 工具取真值 + 富化，能真做到"找到最适合且不幻觉"吗？**

**🟦 正方（支持）**
- 事实取自工具而非模型记忆，是架构性的抗幻觉，比"调教提示"可靠得多。
- 富化把全网脏数据变成可推理知识，多目标排序可解释——这是壁垒。

**🟥 反方（质疑）**
- 富化千万 SKU 的成本与质量是巨大未验证假设，富化差则全盘皆输。
- "最适合"难定义、评测主观；LLM 参与排序则成本高、延迟高。
- 隐性推断易错且可能引入偏见，错一次就伤信任。

**⚖️ 裁决**
- 富化先窄品类做深，用规则 + 品类模板降本；用"需求满足率 + 后悔率"把"最适合"操作化。
- 排序用轻量模型 + 缓存控成本延迟；推断谦逊化、可一键否决。
- **Kill 信号：** 富化质量 / 成本无法规模化，或满足率长期无法超过货架搜索基线。
